## 1. Goal

Define the **v1 user interface** for the ISU Course Manager, scoped to the central plan/overlay view and its supporting interactions. v1 covers everything a student needs to inspect their plan, deviate from the canonical flow, fix downstream cascades, and get contextual AI assistance. The catalog browser, profile, settings, and full wizard-mode (Mode B) are explicitly deferred.

The UI is the React + Vite + TypeScript SPA from the system spec ¬ß3. It speaks to the ASP.NET Core API via the endpoints in the system spec ¬ß6.

**See also:** the 8 mockup HTML files at `docs/superpowers/mockups/ui-v1/` ‚ÄĒ open any in a browser. The two final-state files are `tightened-desktop-v4.html` (locked layout + AI styling) and `interaction-direct-manipulation.html` (locked interaction patterns).

### Wire conventions (used throughout this spec)

- **JSON casing:** all DTO field names are `camelCase` in JSON, mapped to `PascalCase` C# properties on the backend.
- **Enum values:** sent as PascalCase strings matching the C# enum names (`"Planned"`, `"Completed"`, `"Active"`, etc.). Numeric enum codes are not used in JSON.
- **`AcademicTerm` field:** integer in `YYYYSS` form per system spec ¬ß4 Tier 3 ‚ÄĒ `202602` = Fall 2026, `202604` = Spring 2026 (year is the *ending* year of the academic cycle).
- **API base path:** `/api/v1/` ‚ÄĒ every endpoint reference in this spec uses the full prefix.

## 2. Scope

### In scope (v1)

- Plan / overlay view (the central screen) ‚ÄĒ desktop and mobile responsive layouts
- Tile system for rendering `StudentCourse` overlaid on `FlowchartSlot`
- Direct manipulation: click-to-action-menu, immediate moves, drag-and-drop
- Live validation (badges + banner) with `Auto-Fix` for large cascades
- Empty-slot picker (4-section: Recommended / Pull-forward / Catalog / Leave-empty)
- Sidebar: flow switcher, progress stats, AI insights
- AI integration at four scope levels: global / flow / semester / slot
- Top bar with student identity + AI entry point
- Validation banner that adapts to issue count

### Out of scope (deferred)

- Catalog browser screen (separate route, separate spec later)
- Student profile / settings screens
- Wizard Mode B (multi-step batched preview) ‚ÄĒ reachable from "Get help" but not designed in v1
- Multi-level undo (single-level Undo Auto-Fix only)
- Authentication UI (stubbed JWT per system spec)
- Onboarding flows
- Major comparison view as a dedicated screen (overlay panel only for v1)
- Print / Export
- Notifications / inbox

### Accessibility baseline (v1)

Full audit deferred to ¬ß18, but v1 ships with these minimums:

- All tiles render as `<button role="button" aria-label="{course code}, {status}, {grade if any}">`. Enter/Space opens the action menu (same as click).
- Action menu and right panel are keyboard-traversable: Tab cycles, Enter activates, Escape closes the panel.
- Drag-and-drop has a keyboard equivalent: focus a tile, press Enter to open action menu, choose Move-to-future, pick a destination from the inline picker (also keyboard-navigable).
- Banner actions are real `<button>` elements; Auto-Fix all is the first focusable element in the tall variant.
- Color is never the sole indicator of status ‚ÄĒ every status-colored tile also has a text label in its subtitle (e.g., "moved", "in progress", "fail"). Validation flag tiles have both red ring and the `!` badge.
- Contrast ratios meet WCAG AA for body text and tile labels (verified against the chosen palette).

### Mobile (< 768px) ‚ÄĒ single-column kanban with swipe

- Sidebar collapses to a hamburger drawer
- Main area becomes column-per-semester (B-style); user swipes/scrolls horizontally between semesters
- Right panel becomes a bottom sheet that slides up over the main area
- Banner stays at top of main area
- Topbar stays full-width

The data model and component logic are identical between layouts ‚ÄĒ only the outer CSS grid + tile flow direction differ.

## 5. Tile system

Each tile represents a single `StudentCourse` (or unfulfilled `FlowchartSlot`). Compact: 78px min-width √ó 44px min-height, 11px font.

Tile contents:
- Course code (e.g., "Math 1650")
- `<small>` line: grade if completed (e.g., "A ¬∑ 4cr"), or credits + status hint

Tile states (CSS class, mutually exclusive within status):
- `.completed` ‚ÄĒ green tint, optional grade in subtitle
- `.completed.grade-pending` ‚ÄĒ desaturated green tint (`#f1f8e9`) + dashed `#aed581` border + `"grade pending"` italic in subtitle + small ‚Ź≥ amber corner badge top-right. Emitted when `Status=Completed` and `Grade=null/empty`. Added by addendum spec `2026-05-13-pending-grade-and-coreq-cascade-design.md`.
- `.completed.failed-req` ‚ÄĒ pink tint (`#fce4ec`) + solid 1.5px deep-magenta border (`#ad1457`) + `"below req"` red pill top-right. Emitted when a Completed course has at least one downstream `GradeRequirementUnmet` issue with `RelatedStudentCourseId` matching this tile. Derived from validation output, not stored. Added by the same addendum.
- `.inprogress` ‚ÄĒ yellow tint, glowing border
- `.planned` ‚ÄĒ dashed grey border
- `.failed` ‚ÄĒ red tint
- `.unfulfilled-degree` ‚ÄĒ red diagonal stripes; subtitle is one of:
  - `"moved to Sem N"` ‚ÄĒ when this slot was previously filled by a `StudentCourse` whose `AcademicTerm` has changed to a different semester
  - `"unfilled"` ‚ÄĒ when this slot was never filled (e.g., a fresh student exploring a Pending flow with zero StudentCourses)
- `.elective-empty` ‚ÄĒ grey diagonal stripes (placeholder for unfilled elective slot)

Markers (additive to status, can stack):
- `.invalid` ‚ÄĒ red box-shadow ring + small `!` flag badge in top-right corner
- `.invalid.via-coreq` ‚ÄĒ same as `.invalid` but with a small `"co-req broken"` label above the tile (`#c62828` on `#ffebee`, bordered `#ef9a9a`). Emitted when this Planned course was flagged transitively via the Step 2a co-req cascade rather than via its own broken prereq. Added by addendum spec.
- `.pending-dep` ‚ÄĒ yellow dashed ring (`box-shadow: 0 0 0 2px #ffd54f`) + small `"depends on pending grade"` label above the tile (`#fff8e1` bg, `#5a4500` text, `#ffd54f` border). Emitted on Planned tiles with at least one `PendingGradeDependency` warning. Distinct from `.cascade-affected` (drag-preview yellow, solid ring) ‚ÄĒ both can coexist. Added by addendum spec.
- `.pending-coreq-dep` ‚ÄĒ same yellow dashed ring + label `"co-req depends on pending grade"`. Emitted when this Planned course's hard co-req has a `PendingGradeDependency` (indirect risk).
- `.ext-badge` ‚ÄĒ small purple pill (`#6a1b9a` bg, white text, ~8px font, ~1px√ó6px padding) anchored top-left of the tile. Text is the institution's short code (e.g., `"LLCC"`, `"DMACC"`), derived client-side from a small lookup map. Emitted on tiles whose underlying `StudentCourse.EnrollmentSource = External`. Added by addendum spec `2026-05-13-external-transfer-v1-design.md`.
- `.ai-context` ‚ÄĒ small purple sparkle (`‚ú¶`) corner badge in top-right (only one of {invalid, ai-context} at a time on a given tile)
- `.selected` ‚ÄĒ blue box-shadow ring (when clicked / focus)
- `.cascade-affected` ‚ÄĒ yellow box-shadow ring (during drag preview, **solid** ‚ÄĒ distinct from `.pending-dep` which is dashed)
- `.dragging` ‚ÄĒ opacity 0.4, slight rotation
- Auto-Fixed annotation ‚ÄĒ green border on a tile that the engine moved during the most recent Auto-Fix

## 7. Topbar

Three regions, left-to-right:

1. **Brand** ‚ÄĒ `ūüďė ISUCourseManager` with "CourseManager" in gold
2. **AI button (left-of-center)** ‚ÄĒ `‚ú¶ Ask AI` purple gradient pill (un-scoped global AI; opens right panel for general questions)

## 8. Sidebar (left)

Three sections, top-to-bottom:

Each card shows:

- Code ¬∑ Catalog year (e.g., "CYBE ¬∑ 2025-26")

### 8.2 Progress stats

Compact stats card scoped to the currently-viewed flow:

### 8.3 Insights (AI-generated)

Card with subtle purple-accent left borders. Section header: "Insights" + small purple `‚ú¶ AI` capsule.

- One-line action link beneath ("‚Üí Compare flows", "‚Üí Add to Sem 6")

## 9. Validation banner (above main)

Lives in its own row between topbar and main. Conditional. The `Severity` field on each `ValidationIssueDto` (per system spec ¬ß8) determines whether an issue counts as Error or Warning.

| State | Style | Content |
|---|---|---|
| 0 issues | thin green | `‚úď All prereqs validated.` |
| Warnings only (e.g., `PendingGradeDependency`) | thin amber | `‚Ź≥ N grade(s) pending ‚ÄĒ M future course(s) depend on them` (counts include direct + transitive co-req-dep dependents) |
| 1-2 errors | thin red | `! N prereq issues ‚ÄĒ [Show all] [Get help]` |
| 3+ errors | tall red | Bold issue count, explanation, **dominant green "‚§ď Auto-Fix all"** button + secondary "Show all 12" + "Get help (wizard)" |
| Post-Auto-Fix | thin green | Summary of changes (e.g., "Auto-Fixed 12 courses across 5 semesters; graduation pushed to Fall 2030") + `[Looks good] [Undo Auto-Fix]` |

When errors are present, warnings are suppressed in the banner (still visible on tiles). Counts in the error and warning rows include indirect/transitive dependents (e.g., when MATH 1650 fails, the count includes MATH 1660 + PHYS 2310 directly + PHYS 2310L via Step 2a co-req cascade). See addendum `2026-05-13-pending-grade-and-coreq-cascade-design.md` ¬ß7.

The "Get help" button opens the system spec's full wizard (Mode B), which is deferred from v1 design but accessible.

## 10. Right panel (contextual)

Single panel, ~380px wide, becomes one of these based on context (only one shown at a time):

### 10.1 Action menu

Opens when user clicks a course tile. Header card with:
- Breadcrumb: `Sem 1 ¬∑ Fall 2025` for unflagged tiles. Variants for flagged tiles:
  - Single issue: `‚ö† Validation issue ¬∑ Sem 1 ¬∑ Fall 2025`
  - Sibling issue group (multiple issues sharing one root cause via matching `RelatedStudentCourseId`): `‚ö† Issue X of N ¬∑ root cause: [course code + condition] ¬∑ view all`. Clicking "view all" switches the panel to a sibling-group summary view (mini-tile previews of all N affected courses). Added by addendum `2026-05-13-pending-grade-and-coreq-cascade-design.md` ¬ß5.
- Course name + dept + credits + prereq summary
- Status pills (current status, grade, min req); for flagged tiles, an additional `+N affected siblings` pill when sibling group size > 1

Body sections (some conditional):
- **Suggested fix** ‚ÄĒ appears at top when the tile is `.invalid`. Green-tinted card with the most-likely resolution + one-click apply. When the tile is part of a sibling-issue group, the recommended fix is **chain-level** (addresses all N siblings) and listed first; single-tile fixes appear as alternatives. Example chain fix: "Retake MATH 1650 in Sem 3 + defer MATH 1660, PHYS 2310, PHYS 2310L to Sem 4".
- **Update status** ‚ÄĒ Mark Completed (with grade picker), Mark In Progress, Mark Failed/Cancelled (red, danger-styled).
- **Reschedule** ‚ÄĒ "Move to future term" + "Move to earlier term" (each opens an inline destination picker showing valid semesters as buttons; recommended one highlighted green).
- **Replace** ‚ÄĒ "Substitute another course" (opens slot picker scoped to alternatives).
- **Mark as taking externally** ‚ÄĒ purple-styled (`#6a1b9a`). Opens a small form (institution, external course code, expected term, optional note). Saves with `EnrollmentSource=External` and the transfer fields populated; tile re-renders with the institution badge top-left. Added by addendum `2026-05-13-external-transfer-v1-design.md` ¬ß5‚Äď¬ß6.
- **Remove** ‚ÄĒ "Remove from plan" (red).

Footer: `[Close]`.

### 10.2 Slot picker

Opens when user clicks an empty/unfulfilled slot. Header:
- Breadcrumb: `Sem 1 ¬∑ Fall 2025`
- "Fill this slot"
- Context: "Originally: Math 1650 ¬∑ Calc I (now in Sem 2)"

Search input at top of body. Sections:
- **Recommended** ‚ÄĒ single best-fit option (e.g., "‚Ü© Move Math 1650 back here"), green tinted, "Best fit" badge
- **Pull from a later semester** ‚ÄĒ list of courses currently scheduled in later semesters that fit this slot's category. Each shows current vs. target semester with "Pull forward" button.
- **AI suggestions** ‚ÄĒ if relevant for this slot type. Purple-accented suggestion cards with rationale + Add buttons.
- **Add a new course from the catalog** ‚ÄĒ filtered list with course details + prereqs, "Add" buttons.
- **Or‚Ä¶** ‚ÄĒ "Leave this slot empty" with credit consequence noted.

Footer: `[Cancel] [Apply selection]`.

### 10.3 AI panel

Opens via topbar `Ask AI`, header `Analyze flow`, sem-row `‚ú¶`, or "More insights ‚Üí" link. Header:
- `[‚ú¶ AI]` capsule + title (e.g., "Semester analysis" / "Flow analysis" / "General")
- Scope chip on right (e.g., "Sem 2 ¬∑ Spring 2026" or "Whole flow")
- Close button
- Optional meta line ("2 open Gen Ed slots ¬∑ 19 credits planned ¬∑ 4 candidates ranked")

Body:

- **Suggestion cards** for actionable items: course name + meta (cr / offering / prereq), plain-English rationale ("89% of cohort took it"), action buttons:
  - Primary: purple gradient pill ("Add to slot 1")

## 11. AI integration architecture

### Scope levels (entry points)

| Scope | Entry | Use case |
|---|---|---|
| **Global** | Topbar `Ask AI` | "What if I switched majors?" / "Suggest a tech elective" |
| **Flow** | Main header `‚ú¶ Analyze flow` | "How am I doing in this major?" / "Am I close to another?" |
| **Semester** | Sem-row `‚ú¶` | Luke's case: "What Gen Eds fit Sem 2's open slots?" |
| **Slot** | AI section in slot picker | "What course best fits this CybE Elective slot in Sem 6?" |

All four resolve to the same `/api/v1/ai/*` endpoint group with a `scope` parameter.

### Server-mediated (no direct browser ‚Üí Anthropic)

The frontend **never calls Anthropic directly.** All AI requests hit our backend, which:

1. Loads the relevant context (Student, StudentCourses, active StudentDegreeFlows, scope-relevant slice of the Catalog).
2. Constructs a prompt with the spec's domain knowledge + the student's data.
3. Calls Anthropic's Messages API with **prompt caching** enabled (the spec's domain context is cacheable; only the per-request specifics vary).
4. If Claude returns tool-use suggestions ("propose moving X to Sem N"), the backend translates them into action-button payloads in the response. Buttons in the UI execute against our existing `/me/courses` and `/cascade` endpoints ‚ÄĒ Claude doesn't see student auth or modify data directly.
5. Returns the AI message + structured action buttons + suggestion cards to the frontend.

Benefits: auth control, audit logging, prompt-cache hit rates, rate limiting, cost control, all server-side.

### Endpoints (additions to system spec ¬ß6)

```
POST   /api/v1/ai/ask                       ‚Üí AiResponseDto
       body: { scope: { type, semester?, flowId?, slotId? }, prompt?, decisionAnswers? }

GET    /api/v1/me/insights                  ‚Üí AiInsightDto[]    (proactive sidebar insights)
       query: ?flowId=...
```

## 12. Direct manipulation (Mode A)

The default editing pattern. From the system spec ¬ß7 but UI-elaborated here:

### Click a tile

Right panel ‚Üí action menu (¬ß10.1). User picks an action which executes immediately (or opens a sub-flow like the slot picker for "Substitute").

### Move a course to another semester

Two equivalent entry points:
1. **Action menu**: "Move to future term" ‚Üí inline destination picker (semester buttons) ‚Üí click target ‚Üí executes.
2. **Drag-and-drop**: grab a tile, drop on a target semester. While dragging:
   - Source row gets red `cannot drop on origin` border (the *only* invalid drop target enforced visually)
   - Every other semester row gets a green drop-zone border ‚ÄĒ including past semesters
   - Coreq partner tiles get a yellow ring (will be cascade-affected if dropped)
   - A ghost tile follows the cursor with slight rotation

**Drop-target validity:** the UI does NOT pre-validate prereq satisfiability for each candidate semester ‚ÄĒ that would require a chatty `POST /cascade/preview` per drop target during the drag. Drop is permitted on any non-origin row; if the drop creates broken prereqs, post-drop validation flags the moved tile + emits a banner issue. The user iterates from there. (If pre-validation feedback during drag is later wanted, add a lightweight `GET /api/v1/me/cascade/can-place?courseId=&academicTerm=` endpoint per ¬ß17 amendment.)

**Move executes immediately** ‚ÄĒ no preview gate. Validation runs after; broken-prereq tiles get red `.invalid` ring + flag badge; banner appears or updates.

### Click an empty slot

Right panel ‚Üí slot picker (¬ß10.2). User picks a course (or moves one forward, or leaves empty). Pick executes immediately.

### Iterative validation flow

1. User makes a move; validation banner shows N issues
2. Click any flagged tile (or the banner's "Show all"); action menu opens with `Suggested fix` card at top
3. User accepts the suggested fix or picks an alternative
4. Validation re-runs; banner updates
5. When 0 issues, banner turns green

### Auto-Fix (large cascades)

When the cascade engine reports 3+ broken-prereq issues from a single move, the validation banner becomes the "tall" variant with **`‚§ď Auto-Fix all`** as the dominant green action. One click runs the cascade engine: every broken course is moved to its `earliestValidSemester`. Result:

- Banner turns green: "Auto-Fixed 12 courses across 5 semesters. Projected graduation pushed 1 semester (Spring 2030 ‚Üí Fall 2030). 4 empty slots opened up ‚ÄĒ click to fill."
- Tiles moved by the engine get a green ring + `‚§ī Auto-Fixed` annotation in the subtitle
- New empty slots from the moves get the unfulfilled-degree style
- A new semester appears at the end if any course was pushed past Sem 8
- `[Looks good]` and `[Undo Auto-Fix]` buttons in the banner

#### Auto-Fix wire protocol

The system spec's cascade engine takes a `CascadeTrigger` and produces a `CascadeProposal` with `Decisions` the wizard normally surfaces to the user. Auto-Fix bypasses user interaction by *auto-answering* every decision with the engine's default. The protocol:

1. Frontend already has the most recent move's trigger in memory (`CourseAddedToTerm` from a drag, `CourseRemovedFromTerm` from a delete, etc.). If multiple moves accumulated since the last `/cascade/apply`, the frontend has the trigger set for the most recent only ‚ÄĒ preceding moves were already validated standalone.
2. **`POST /api/v1/cascade/preview`** with that trigger and `decisionAnswers: []`. Inspect the response.
3. For every `decisions[]` entry, build an auto-answer using the engine's recommended default (e.g., for `FillGapDecision`, accept the suggested slot list; for `ConfirmOverloadDecision`, accept the overload as-is). The system spec's `DecisionPoint` types each define a recommended default.
4. **`POST /api/v1/cascade/preview`** again with the accumulated `decisionAnswers[]`. Repeat until `decisions` is empty (typically one or two iterations).
5. **`POST /api/v1/cascade/apply`** with the same trigger + final `decisionAnswers[]`. Backend commits in a transaction and returns the changed `StudentCourseDto[]` plus any affected `StudentDegreeFlowDto[]`.

If the engine returns no broken state for the given trigger (i.e., the user clicked Auto-Fix when there were ambient `ValidationIssueDto` warnings unrelated to the most recent move), the frontend instead uses a synthesized **`FixCurrentState` trigger** (new ‚ÄĒ see ¬ß17): `POST /cascade/preview` with `{ trigger: { type: "FixCurrentState" } }` and the same auto-answer loop. The cascade engine treats this as "pretend every currently-broken `StudentCourse` was just placed there as a trigger and re-derive valid placements."

#### Undo Auto-Fix

`Undo Auto-Fix` is single-level only and only valid until the next mutation.

Implementation is **client-side**: immediately before calling `/cascade/apply` for an Auto-Fix run, the frontend snapshots the current `StudentCourse[]` from `useStudent()` state into a sessionStorage key. On `[Undo Auto-Fix]` click, the frontend replays the snapshot via **`POST /api/v1/me/courses/restore`** (new ‚ÄĒ see ¬ß17), a bulk endpoint that takes a `StudentCourseDto[]` and replaces the student's current courses with the snapshot in a single transaction. The snapshot key is cleared on any subsequent mutation to disable the Undo button.

Avoiding server-side undo storage keeps the system spec's "no undo" stance intact while still giving the user a one-click escape hatch from a regrettable Auto-Fix.
