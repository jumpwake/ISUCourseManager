# ISU Course Manager — UI v1 Design Spec

**Date:** 2026-05-13
**Status:** Approved for implementation planning
**Author:** Kevin (with Claude)
**Companion to:** `docs/superpowers/specs/2026-05-12-isu-course-manager-design.md` (system design / domain model / API)

## 1. Goal

Define the **v1 user interface** for the ISU Course Manager, scoped to the central plan/overlay view and its supporting interactions. v1 covers everything a student needs to inspect their plan, deviate from the canonical flow, fix downstream cascades, and get contextual AI assistance. The catalog browser, profile, settings, and full wizard-mode (Mode B) are explicitly deferred.

The UI is the React + Vite + TypeScript SPA from the system spec §3. It speaks to the ASP.NET Core API via the endpoints in the system spec §6.

## 2. Scope

### In scope (v1)

- Plan / overlay view (the central screen) — desktop and mobile responsive layouts
- Tile system for rendering `StudentCourse` overlaid on `FlowchartSlot`
- Direct manipulation: click-to-action-menu, immediate moves, drag-and-drop
- Live validation (badges + banner) with `Auto-Fix` for large cascades
- Empty-slot picker (4-section: Recommended / Pull-forward / Catalog / Leave-empty)
- Sidebar: flow switcher, progress stats, AI insights
- AI integration at four scope levels: global / flow / semester / slot
- Top bar with student identity + AI entry point
- Validation banner that adapts to issue count

### Out of scope (deferred)

- Catalog browser screen (separate route, separate spec later)
- Student profile / settings screens
- Wizard Mode B (multi-step batched preview) — reachable from "Get help" but not designed in v1
- Multi-level undo (single-level Undo Auto-Fix only)
- Authentication UI (stubbed JWT per system spec)
- Onboarding flows
- Major comparison view as a dedicated screen (overlay panel only for v1)
- Print / Export
- Notifications / inbox

```
│          │  ─ banner (validation, conditional)  │ (contextual:  │

```

CSS grid: `grid-template-columns: 240px 1fr 380px`. The right panel collapses (`grid-template-columns: 240px 1fr`) when no contextual content is active.

### Mobile (< 768px) — single-column kanban with swipe

- Sidebar collapses to a hamburger drawer
- Main area becomes column-per-semester (B-style); user swipes/scrolls horizontally between semesters
- Right panel becomes a bottom sheet that slides up over the main area
- Banner stays at top of main area
- Topbar stays full-width

The data model and component logic are identical between layouts — only the outer CSS grid + tile flow direction differ.

## 5. Tile system

Each tile represents a single `StudentCourse` (or unfulfilled `FlowchartSlot`). Compact: 78px min-width × 44px min-height, 11px font.

Tile contents:
- Course code (e.g., "Math 1650")
- `<small>` line: grade if completed (e.g., "A · 4cr"), or credits + status hint

Tile states (CSS class, mutually exclusive within status):
- `.completed` — green tint, optional grade in subtitle
- `.inprogress` — yellow tint, glowing border
- `.planned` — dashed grey border
- `.failed` — red tint
- `.unfulfilled-degree` — red diagonal stripes ("⚠ Math 1650 — moved to Sem 2")
- `.elective-empty` — grey diagonal stripes (placeholder for unfilled elective slot)

Markers (additive to status, can stack):
- `.invalid` — red box-shadow ring + small `!` flag badge in top-right corner
- `.ai-context` — small purple sparkle (`✦`) corner badge in top-right (only one of {invalid, ai-context} at a time on a given tile)
- `.selected` — blue box-shadow ring (when clicked / focus)
- `.cascade-affected` — yellow box-shadow ring (during drag preview)
- `.dragging` — opacity 0.4, slight rotation
- Auto-Fixed annotation — green border on a tile that the engine moved during the most recent Auto-Fix

## 7. Topbar

Three regions, left-to-right:

1. **Brand** — `📘 ISUCourseManager` with "CourseManager" in gold
2. **AI button (left-of-center)** — `✦ Ask AI` purple gradient pill (un-scoped global AI; opens right panel for general questions)

## 8. Sidebar (left)

Three sections, top-to-bottom:

### 8.1 My degree flows

List of `StudentDegreeFlow` cards, sorted: Active first, then Pending, Completed and Deleted hidden by default.

Each card shows:
- Flow name (e.g., "Cyber Security Engr")
- Code · Catalog year (e.g., "CYBE · 2025-26")
- Progress bar with % and `28/125` cr counter
- Status pill: `Active` (blue), `Pending` (amber + dashed border)
- "👁 viewing" indicator when this flow is the one rendered in main

A **"+ Test another flow"** card at the bottom — same shape as the flow cards but with dashed grey border and label "+ Test another flow / Add a Pending flow to overlay against your courses." Click → opens flow picker (deferred screen).

### 8.2 Progress stats

Compact stats card scoped to the currently-viewed flow:
- Credit total + percentage with progress bar
- Status counts grid: completed / in-progress / planned / warnings
- Projected graduation term

### 8.3 Insights (AI-generated)

Card with subtle purple-accent left borders. Section header: "Insights" + small purple `✦ AI` capsule.

Each insight is a clickable item with:
- Plain-English observation (one or two sentences)
- One-line action link beneath ("→ Compare flows", "→ Add to Sem 6")

The first insight ranked by priority gets a deeper purple background. "More insights →" link at the bottom opens the un-scoped AI chat.

## 9. Validation banner (above main)

Lives in its own row between topbar and main. Conditional:

| State | Style | Content |
|---|---|---|
| 0 issues | thin green | `✓ All prereqs validated.` |
| 1-2 issues | thin red | `! N prereq issues — [Show all] [Get help]` |
| 3+ issues | tall red | Bold issue count, explanation, **dominant green "⤓ Auto-Fix all"** button + secondary "Show all 12" + "Get help (wizard)" |
| Post-Auto-Fix | thin green | Summary of changes (e.g., "Auto-Fixed 12 courses across 5 semesters; graduation pushed to Fall 2030") + `[Looks good] [Undo Auto-Fix]` |

The "Get help" button opens the system spec's full wizard (Mode B), which is deferred from v1 design but accessible.

## 10. Right panel (contextual)

Single panel, ~380px wide, becomes one of these based on context (only one shown at a time):

### 10.1 Action menu

Opens when user clicks a course tile. Header card with:
- Breadcrumb: `Sem 1 · Fall 2025` (or `⚠ Validation issue · Sem 1 · Fall 2025` for flagged tiles)
- Course name + dept + credits + prereq summary
- Status pills (current status, grade, min req)

Body sections (some conditional):
- **Suggested fix** — appears at top when the tile is `.invalid`. Green-tinted card with the most-likely resolution + one-click apply. Example: "Move CprE 1850 to Sem 2 (with Math 1650)".
- **Update status** — Mark Completed (with grade picker), Mark In Progress, Mark Failed/Cancelled (red, danger-styled).
- **Reschedule** — "Move to future term" + "Move to earlier term" (each opens an inline destination picker showing valid semesters as buttons; recommended one highlighted green).
- **Replace** — "Substitute another course" (opens slot picker scoped to alternatives).
- **Remove** — "Remove from plan" (red).

Footer: `[Close]`.

### 10.2 Slot picker

Opens when user clicks an empty/unfulfilled slot. Header:
- Breadcrumb: `Sem 1 · Fall 2025`
- "Fill this slot"
- Context: "Originally: Math 1650 · Calc I (now in Sem 2)"

Search input at top of body. Sections:
- **Recommended** — single best-fit option (e.g., "↩ Move Math 1650 back here"), green tinted, "Best fit" badge
- **Pull from a later semester** — list of courses currently scheduled in later semesters that fit this slot's category. Each shows current vs. target semester with "Pull forward" button.
- **AI suggestions** — if relevant for this slot type. Purple-accented suggestion cards with rationale + Add buttons.
- **Add a new course from the catalog** — filtered list with course details + prereqs, "Add" buttons.
- **Or…** — "Leave this slot empty" with credit consequence noted.

Footer: `[Cancel] [Apply selection]`.

### 10.3 AI panel

Opens via topbar `Ask AI`, header `Analyze flow`, sem-row `✦`, or "More insights →" link. Header:
- `[✦ AI]` capsule + title (e.g., "Semester analysis" / "Flow analysis" / "General")
- Scope chip on right (e.g., "Sem 2 · Spring 2026" or "Whole flow")
- Close button
- Optional meta line ("2 open Gen Ed slots · 19 credits planned · 4 candidates ranked")

Body:
- AI message blocks (flat typography, no chat bubbles) with bold purple emphasis on key facts
- **Suggestion cards** for actionable items: course name + meta (cr / offering / prereq), plain-English rationale ("89% of cohort took it"), action buttons:
  - Primary: purple gradient pill ("Add to slot 1")
  - Ghost: "Why?", "Show alternatives"
  - Recommended cards prefixed with `✦` and have purple wash
- **Tip cards** for batch suggestions: e.g., "If you accept PSYCH 2300 + PHIL 2300, your Sem 2 stays at 19cr — Apply both"
- "Show N more →" expander when more suggestions exist

Footer:
- Free-text input ("Ask about Sem 2…" — placeholder reflects scope)
- Purple gradient `Ask` send button
- Quick-ask chips (3-4): pre-built prompts relevant to the scope

## 11. AI integration architecture

### Scope levels (entry points)

| Scope | Entry | Use case |
|---|---|---|
| **Global** | Topbar `Ask AI` | "What if I switched majors?" / "Suggest a tech elective" |
| **Flow** | Main header `✦ Analyze flow` | "How am I doing in this major?" / "Am I close to another?" |
| **Semester** | Sem-row `✦` | Luke's case: "What Gen Eds fit Sem 2's open slots?" |
| **Slot** | AI section in slot picker | "What course best fits this CybE Elective slot in Sem 6?" |

All four resolve to the same `/api/v1/ai/*` endpoint group with a `scope` parameter.

### Server-mediated (no direct browser → Anthropic)

The frontend **never calls Anthropic directly.** All AI requests hit our backend, which:

1. Loads the relevant context (Student, StudentCourses, active StudentDegreeFlows, scope-relevant slice of the Catalog).
2. Constructs a prompt with the spec's domain knowledge + the student's data.
3. Calls Anthropic's Messages API with **prompt caching** enabled (the spec's domain context is cacheable; only the per-request specifics vary).
4. If Claude returns tool-use suggestions ("propose moving X to Sem N"), the backend translates them into action-button payloads in the response. Buttons in the UI execute against our existing `/me/courses` and `/cascade` endpoints — Claude doesn't see student auth or modify data directly.
5. Returns the AI message + structured action buttons + suggestion cards to the frontend.

Benefits: auth control, audit logging, prompt-cache hit rates, rate limiting, cost control, all server-side.

### Endpoints (additions to system spec §6)

```
POST   /api/v1/ai/ask                       → AiResponseDto
       body: { scope: { type, semester?, flowId?, slotId? }, prompt?, decisionAnswers? }

GET    /api/v1/me/insights                  → AiInsightDto[]    (proactive sidebar insights)
       query: ?flowId=...
```

`AiResponseDto` shape (sketch):

```jsonc
{
  "messages": [
    { "kind": "text", "content": "Sem 2 is already 19 credits …", "highlights": ["19 credits"] },
  ],
  "suggestions": [
    {
      "courseId": "PSYCH-2300",
      "name": "PSYCH 2300 · Intro to Psychology",
      "meta": "3 cr · F/S/Su · no prereq",
      "rationale": "Most-picked Gen Ed among CybE majors (89% of cohort) …",
      "isRecommended": true,
      "actions": [
        { "label": "Add to slot 1", "kind": "primary",
          "execute": { "endpoint": "POST /me/courses",
                       "body": { "courseId": "PSYCH-2300", "academicTerm": 202604, "status": "Planned" } } }
      ]
    }
  ],
  "tip": { "text": "If you accept PSYCH 2300 + PHIL 2300, …",
           "action": { "label": "Apply both", "execute": { ... } } },
  "scope": { "type": "Semester", "semester": 2, "academicTerm": 202604 },
  "tokens": { "in": 1240, "out": 380 },
  "latencyMs": 380
}
```

## 12. Direct manipulation (Mode A)

The default editing pattern. From the system spec §7 but UI-elaborated here:

### Click a tile

Right panel → action menu (10.1). User picks an action which executes immediately (or opens a sub-flow like the slot picker for "Substitute").

### Move a course to another semester

Two equivalent entry points:
1. **Action menu**: "Move to future term" → inline destination picker (semester buttons) → click target → executes.
2. **Drag-and-drop**: grab a tile, drop on a target semester. While dragging:
   - Source row gets red `cannot drop on origin` border
   - Valid target rows get green border + drop hint
   - Other rows get lighter green "also valid"
   - Coreq partner tiles get yellow ring (will be cascade-affected if dropped)
   - A ghost tile follows the cursor with rotation

**Move executes immediately** — no preview gate. Validation runs after; broken-prereq tiles get red `.invalid` ring + flag badge; banner appears or updates.

### Click an empty slot

Right panel → slot picker (10.2). User picks a course (or moves one forward, or leaves empty). Pick executes immediately.

### Iterative validation flow

1. User makes a move; validation banner shows N issues
2. Click any flagged tile (or the banner's "Show all"); action menu opens with `Suggested fix` card at top
3. User accepts the suggested fix or picks an alternative
4. Validation re-runs; banner updates
5. When 0 issues, banner turns green

### Auto-Fix (large cascades)

When the cascade engine reports 3+ broken-prereq issues from a single move, the validation banner becomes the "tall" variant with **`⤓ Auto-Fix all`** as the dominant green action. One click runs the cascade engine: every broken course is moved to its `earliestValidSemester`. Result:

- Banner turns green: "Auto-Fixed 12 courses across 5 semesters. Projected graduation pushed 1 semester (Spring 2030 → Fall 2030). 4 empty slots opened up — click to fill."
- Tiles moved by the engine get a green ring + `⤴ Auto-Fixed` annotation in the subtitle
- New empty slots from the moves get the unfulfilled-degree style
- A new semester appears at the end if any course was pushed past Sem 8
- `[Looks good]` and `[Undo Auto-Fix]` buttons in the banner

`Undo Auto-Fix` is single-level only and only valid until the next mutation.

## 13. Acceptance criteria for v1 UI

These are the user-visible behaviors v1 must satisfy. Each becomes a Playwright E2E test scenario or a Storybook visual snapshot.

**Layout:**
- **UI-AC-1** Plan view renders all 8 semesters with sem-label + credits + tiles, scrolling vertically as needed.
- **UI-AC-2** At ≥768px viewport, layout is three-column (sidebar / main / panel). At <768px, sidebar collapses to drawer and main becomes column-per-semester swipeable.
- **UI-AC-3** Right panel collapses when no tile selected and no AI panel open.

**Tile rendering:**
- **UI-AC-4** Each tile shows course code, status color, dept tint, grade-if-completed, credits.
- **UI-AC-5** A `StudentCourse` with `Status=Completed` and `Grade="A"` renders with green completed style, "A" in subtitle.
- **UI-AC-6** An `unfulfilled-degree` slot (a `FlowchartSlot.SlotType=DegreeClass` with no matching `StudentCourse`) renders with red diagonal stripes and "⚠ classCode — moved" subtitle.
- **UI-AC-7** A flagged tile (validation issue) shows a red ring + `!` badge in top-right corner.

**Direct manipulation:**
- **UI-AC-8** Clicking a tile opens the right-panel action menu with the tile's context.
- **UI-AC-9** Selecting "Move to future" → picking a semester → executes the move and closes the menu within one client-server round-trip.
- **UI-AC-10** Drag-and-drop a tile onto another semester executes the move equivalently to the menu path.
- **UI-AC-11** After a move, the validation banner updates to reflect the new issue count.

**Empty slot picker:**
- **UI-AC-12** Clicking an unfulfilled-degree tile opens the slot picker (10.2) with all 4 sections populated as appropriate.
- **UI-AC-13** "Pull from a later semester" lists only courses currently scheduled in later semesters that match the slot's category.
- **UI-AC-14** Picking an option immediately fills the slot and closes the picker.

**Validation banner:**
- **UI-AC-15** With 0 issues, banner is green: "✓ All prereqs validated."
- **UI-AC-16** With 1-2 issues, banner is thin red with "Show all" + "Get help" actions.
- **UI-AC-17** With 3+ issues, banner is tall red with `⤓ Auto-Fix all` as primary green CTA.
- **UI-AC-18** Clicking "Auto-Fix all" runs the cascade engine; tiles moved by the engine show the green ring + `⤴ Auto-Fixed` annotation; banner turns green with summary; "Undo Auto-Fix" available.

**Sidebar:**
- **UI-AC-19** Flow switcher shows all of the student's `StudentDegreeFlow`s where `Status ∈ {Active, Pending}`, sorted Active first.
- **UI-AC-20** Currently-viewed flow has the "👁 viewing" indicator and active blue border.
- **UI-AC-21** Clicking another flow card switches the main view to that flow.
- **UI-AC-22** "+ Test another flow" card opens the flow picker (deferred; can be a placeholder modal in v1).
- **UI-AC-23** Insights card shows up to 3 AI-generated observations from `/api/v1/me/insights`; clicking an insight executes its action.

**AI integration:**
- **UI-AC-24** Topbar `Ask AI` opens the right panel as the AI chat with global scope.
- **UI-AC-25** Main header `Analyze flow` opens the AI panel with flow scope; scope chip in panel header reads the flow name.
- **UI-AC-26** Sem-row `✦` button opens the AI panel scoped to that semester; scope chip reads "Sem N · Term YYYY"; tiles in that semester get AI-context corner sparkles for as long as the panel is scoped there.
- **UI-AC-27** AI suggestion cards render rationale + action buttons; clicking a primary action executes the underlying API call (e.g., `POST /me/courses`) and surfaces the resulting plan change in the main view.
- **UI-AC-28** Tip cards with batch actions execute multiple changes in one transaction.
- **UI-AC-29** AI quick-ask chips submit the chip text as the next prompt without requiring the user to type.

## 15. State management (sketch)

- `useStudent()` — current student + StudentCourses + StudentDegreeFlows; cached
- `useActiveFlow()` — currently-viewed flow id; URL-synced
- `useValidation()` — derived: runs on every `useStudent()` change; provides issue count + per-tile flags
- `useRightPanel()` — what's currently in the right panel: `null | 'action-menu' | 'slot-picker' | 'ai-panel'` + payload
- `useAi(scope)` — manages an AI conversation scoped to a level; returns messages + suggestions + send
- `useDragDrop()` — drag source, valid targets, ghost position

State stays in React context / `useState` for v1; introduce Zustand or Redux if we hit prop-drilling pain in v2.

## 16. Decisions log (UI-specific)

| # | Decision | Why |
|---|---|---|
| 1 | Responsive: rows-on-desktop / columns-on-mobile | Desktop matches printed CybE chart; mobile column swipe is the only sane phone pattern |
| 2 | Three-column desktop layout (sidebar / main / panel) | Sidebar = always-on context (flows, stats, insights); main = the work; panel = task-focused, mutually exclusive |
| 3 | Right panel is mutually exclusive | Action menu, slot picker, and AI chat share one slot — keeps screen real estate; only one task focus at a time |
| 4 | Move course = immediate, no preview gate | Spec's Mode A — direct manipulation feels lighter; validation runs after and flags issues |
| 5 | Banner adapts to cascade size | 1-2 issues = manual fix is cheap; 3+ = manual is exhausting, surface Auto-Fix as the primary action |
| 6 | `Auto-Fix` (renamed from "Auto-shuffle") | More obvious to users what it does |
| 7 | AI uses purple gradient + `✦` sparkle | Distinct from semantic colors (red/blue/green/yellow) — never confused with status; visible AI-vs-system distinction |
| 8 | AI scoped at four levels (global/flow/semester/slot) | Most useful AI is contextual; Luke's "what Gen Eds fit Sem 2" is a semester-scoped question |
| 9 | AI mediated by our server, never direct from browser | Auth, audit, prompt caching, rate limits, cost — all stay in our control |
| 10 | Insights box surfaces proactive AI observations | Not all AI use is reactive; some observations should be visible without asking |
| 11 | AI suggestion cards include primary + ghost actions + rationale | Suggestions need to be actionable in one click and explainable |
| 12 | Empty slot picker has 4 sections, with "Pull from later" as a first-class option | Matches Luke's actual flow (he pulled GenEds forward from Sems 5/6) |
| 13 | Per-row credit total below the term label, color-coded by threshold | Always-visible workload indicator; spots over-credit semesters at a glance |
| 14 | Flow cards in sidebar show progress bar + status pill | Multi-flow students see at a glance how each is progressing |
| 15 | "Test another flow" presented as a card matching the existing flow cards | Consistent visual language; affords what-if exploration as first-class |

## 17. What this spec enables / requires from the backend

Cross-references to system spec; flag what UI v1 needs that may require backend additions.

**Required (probably already in system spec):**
- `GET /me/flows` → list of StudentDegreeFlow with status filter
- `GET /me/courses` → all StudentCourses
- `GET /me/flows/{id}/overlay` → rendered fit (slot fill map + unfilled + surplus + completion%)
- `POST /me/courses`, `PUT /me/courses/{id}`, `DELETE /me/courses/{id}` — direct manipulation
- `POST /cascade/preview` and `/cascade/apply` — used by Auto-Fix and Replan-from-here

**New backend work surfaced by UI v1:**
- `POST /api/v1/ai/ask` — scoped AI requests (4 scope types)
- `GET /api/v1/me/insights` — proactive sidebar insights endpoint
- A new `AiService` in the backend that constructs prompts, calls Anthropic with prompt caching, parses tool-use into action button payloads
- Server needs to know the 12-credit soft min and 18-credit soft max to color-code per-row credits — `CascadeOptions` in the system spec already has these
- Validation issues need to expose enough structure for the UI's tile-flag + suggested-fix flow — system spec's `ValidationIssueDto` should suffice

These additions should fold into the next backend plan revision.
